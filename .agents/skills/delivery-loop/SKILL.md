# Delivery Loop - 真实交付框架

## 核心理念

让 agent 以**可重复的机制**持续推进交付物（docs/code/assets），并证明进度，而不是写计划或"盘点"。

## 双循环机制

### 1. Work Loop (快节奏，例如每 15 分钟)
- 执行实际工作，改变交付物
- **必须**产生可验证的产出（git commit、更新文件、导出文档等）
- 聚焦于**产出**而不是计划

### 2. Report Loop (慢节奏，例如每小时)
- **只总结**实际交付的内容
- **必须**引用证据（commit ID / 变更路径 / URL）
- 如果没有证据，报告必须说"无进度 + 原因 + 下一步具体交付物"

## 反作弊规则（"Anti Fake-Progress" Gates）

### Rule A - 报告必须基于证据
- 报告必须包含 commit ID 或具体的产出物列表
- 如果上一个窗口内没有证据，报告必须说"无进度 + 原因 + 下一步具体交付物"
- **绝不允许**用"盘点/计划"代替实际交付

## 工作流程

1. **启动工作循环**
   - 接收任务或目标
   - 确定第一个交付物（commit 或文件）

2. **执行工作**
   - 实际编码/写作/修改
   - 提交到 git 或更新文件
   - 生成可验证的产出

3. **记录证据**
   - 记录 commit ID
   - 记录变更的文件路径
   - 记录生成的产出物

4. **生成报告**
   - 只报告实际交付的内容
   - 引用证据（commit ID、文件路径）
   - 如果无产出，说明原因和下一步计划

## 使用场景

- 用户要求每小时/每天的进度报告
- 用户说"进度感觉是假的"或"需要可重复的推进机制"
- 需要持续交付文档、代码、资源等

## 验证清单

在生成报告前，必须检查：
- [ ] 有实际的产出物吗？（commit、文件更新、导出文档）
- [ ] 报告中包含证据吗？（commit ID、路径、URL）
- [ ] 没有把计划当作实际交付吗？
- [ ] 如果无产出，说明了原因和下一步吗？

## 示例

**正确的报告**：
```
过去 15 分钟的交付：
- commit 3a2b1c9: 更新 AGENTS.md，添加任务执行优先级
- 生成 HEARTBEAT.md：定时任务配置
- 更新 memory/2026-03-26.md：记录会话内容
```

**错误的报告**（禁止）：
```
过去 15 分钟的计划：
- 计划更新 AGENTS.md
- 计划生成 HEARTBEAT.md
- 计划记录会话内容
```
